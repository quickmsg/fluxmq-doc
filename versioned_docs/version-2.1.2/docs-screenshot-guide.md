# 文档中的界面截图说明

## 关于「系统动态截图」

**当前文档环境无法自动对运行中的 FluxMQ 控制台做动态截图**（无法启动浏览器、无法访问本地运行的服务）。因此界面类文档需要依赖以下方式配合完成：

1. **手动截图**：在本地或测试环境启动 FluxMQ，对控制台/规则引擎/数据源等页面截图，再按下方规范放入文档。
2. **自动化截图（可选）**：在 CI 或本地用 Playwright / Puppeteer 等工具启动服务并打开页面后截图，将生成图片放入 `static/images` 并更新对应 Markdown。

下面说明如何在文档中**引用已有截图**以及**新增截图**的规范，便于后续你补充界面图时保持风格一致。

## 图片存放位置

- 根目录：`fluxmq-doc/static/images/`
- 建议按模块分子目录，例如：
  - `static/images/config/` — 配置相关
  - `static/images/function/` — 功能（规则引擎、会话、延迟等）
  - `static/images/endpoint/` — 端点管理
  - `static/images/acl/` — ACL
  - `static/images/command/` — 指令消费

## 在 Markdown 中引用截图

Docusaurus 中推荐使用 `@site` 引用站点静态资源：

```markdown
![配置页说明](@site/static/images/config/read.png)
```

- 文件名尽量见名知意（如 `rule-list.png`、`datasource-mysql.png`）。
- 多张图时可在同一目录下用数字或功能前缀区分（如 `endpoint-1.png`、`endpoint-2.png`）。

## 编写「带截图的」功能文档的推荐步骤

1. **先写正文**：在对应版本文档（如 `version-2.1.2`）下写好功能说明、配置项、操作步骤等。
2. **预留截图位置**：在需要插图处插入占位，例如：
   ```markdown
   <!-- 截图：规则列表页 -->
   ![规则列表](@site/static/images/function/rule-list.png)
   ```
3. **本地或 CI 截图**：启动 FluxMQ，打开对应页面，截取界面后保存为上述路径的文件名。
4. **校验**：本地执行 `npm run start` 查看文档站点，确认图片显示正常。

## 如需自动化截图（可选）

若希望「每次发版或构建时自动更新界面截图」，可以在仓库中增加一个脚本，在 CI 中：

1. 启动 FluxMQ（或使用已有测试环境地址）。
2. 使用 Playwright 打开浏览器，访问控制台 URL，按页面执行登录、跳转等操作。
3. 对目标区域或全页执行 `page.screenshot()`，保存到 `static/images/` 下对应路径。
4. 提交生成的图片或作为构建产物发布。

这样即可在**不依赖人工每次手动截图**的前提下，让文档中的「系统截图」随版本更新而更新，实现近似「动态截图」的效果。

---

**总结**：文档侧无法直接对系统做动态截图，但通过「手动截图 + 规范路径与引用」或「Playwright 等自动化截图脚本」，即可为 2.1.2 及后续版本编写并维护带截图的界面说明文档。若你提供具体页面或模块名称，我可以按上述规范直接写好对应 Markdown 结构和占位图引用，你只需补图即可。
